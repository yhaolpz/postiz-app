## 完整旁白脚本

### 时间轴自动生成｜前言

很多团队做 Agent 的第一步，就是选框架、画多 Agent 架构，再让几个模型互相讨论。

但 Anthropic 在参与大量真实项目后发现，效果最好的系统，往往使用的是更简单、更清晰、更容易组合的模式。真正决定系统质量的，通常不是 Agent 的数量，而是任务边界、评价标准和环境反馈是否清楚。

复杂架构不一定让任务完成得更好，却一定会增加模型调用、延迟、成本和排查难度。每增加一次模型接力，还会多出一次上下文丢失、格式偏差和错误继续传播的机会。

这期视频会帮你解决三个最容易选错的问题：什么时候一次模型调用就够了，五种常见 Workflow 分别适合什么任务，以及什么时候才值得升级为自主 Agent。

看完以后，你会得到一套可以直接用于项目的架构判断顺序，避免为了看起来更像 Agent，付出不必要的复杂度。

### 时间轴自动生成｜架构判断

先区分两个经常被混在一起的概念。

Workflow 的执行路径由代码提前规定。模型可以参与某个步骤，但整体流程仍由程序控制。

Agent 则会根据当前结果，自己判断下一步做什么、调用哪个工具，以及什么时候结束任务。

例如，一份报销单永远需要经过信息提取、规则检查和审批，这条路径可以提前画出来，更适合 Workflow。

但排查线上故障时，你无法提前知道问题在哪。模型可能先读日志，再查监控，然后根据结果决定检查数据库、网络还是代码。这种任务才更适合 Agent。

两者没有谁更高级。Workflow 追求稳定和可预测，Agent 追求灵活和动态决策。

整篇文章最重要的原则是：先寻找最简单的可行方案，只在结果确实不够好时增加复杂度。这里的“结果不够好”，不是主观觉得架构不够先进，而是准确率、完成率、耗时或人工介入次数没有达到目标。先把问题量化，才知道新增的一层编排到底有没有价值。一个实用的判断是：如果复杂方案只是让流程看起来更完整，却没有让同一批测试任务更稳定地完成，它就不是升级，而是负担。

小结。

第一，固定路径优先使用 Workflow。

第二，只有动态路径才真正需要 Agent。

第三，新增复杂度必须换来可测量的结果。

### 时间轴自动生成｜基础单元

在 Workflow 和 Agent 之前，还有一个更基础的单元：增强型大模型。

它不是多个 Agent，而是一个模型接入检索、工具和记忆。

检索让模型获得当前需要的外部事实，而不是只依赖训练数据。工具让模型能够查询订单、读取数据库、调用 API，或者真正执行操作。记忆保存跨步骤有价值的状态，让系统不必每次都从零开始。这三种能力分别解决知道什么、能做什么和已经做过什么，不要把它们混成一个模糊的万能工具。

例如，一个售后助手可以检索退款规则，查询用户订单，调用退款接口，同时记住用户已经确认过的信息。如果订单查询失败，它应该把接口错误返回给模型，而不是编造一个订单状态；如果退款已经执行过，记忆和接口幂等性还要阻止重复退款。

这已经能够解决大量真实任务，并不一定需要多 Agent。

关键不是接入的能力越多越好，而是这些能力是否适合当前任务，接口是否清楚，以及模型能不能正确理解它们应该在什么时候使用。先让一个增强型大模型在受控任务上跑通，通常比过早拆成多个角色，更容易看清真正缺少的是知识、行动能力，还是决策能力。

小结。

第一，检索负责补充外部事实。

第二，工具负责执行真实行动。

第三，记忆负责保留任务状态。

### 时间轴自动生成｜工作流模式

当一次模型调用不够，但任务路径仍然可以控制时，可以从五种 Workflow 中选择。

第一种是提示链，也就是把复杂任务拆成连续的小步骤。

例如先生成文章大纲，再检查大纲是否符合要求，最后根据通过检查的大纲写正文。每一步只解决一个明确问题，中间还可以加入程序化检查。某一步不合格，就不继续向后执行。

它适合能够被清晰拆分的任务。代价是调用次数增加、速度变慢，但每一步更简单，结果也更容易检查。最关键的是给步骤之间设置门槛：大纲缺少目标读者就退回重写，事实校验没有通过就不进入发布，而不是让错误一路传到最后。

第二种是路由。

路由先判断输入属于哪一类，再把它交给对应的提示词、模型或工具。客服问题可以被分成产品咨询、退款申请和技术故障；普通问题交给成本更低的模型，复杂问题交给能力更强的模型。

当不同输入需要明显不同的处理方式，并且分类足够可靠时，路由通常比一个万能 Agent 更简单，也更容易优化。路由还需要处理低置信度输入，例如分类不确定时交给通用流程或人工，而不是强行送入一个错误分支。

第三种是并行，它有两种用法。

任务分块，是把互不依赖的子任务同时执行，再汇总结果。例如评估一个 AI 回答时，可以同时检查事实准确性、安全性、表达质量和格式要求。

多次投票，则是让多个模型独立判断同一个问题，再聚合结果。例如审查代码漏洞时，可以让多个模型从不同角度检查，达到一定票数后再判定风险。

并行适合两种情况：要么子任务彼此独立，可以并发处理；要么单次判断不够可靠，需要多个独立视角提高信心。多个判断只有在信息、提示词或关注角度真正独立时才有价值；把同一个提示词复制三遍，往往只会得到三份相似偏差。

第四种是编排者与执行者。

一个中央模型先理解任务，动态拆分子任务，再交给多个执行模型处理，最后汇总结果。

它和并行的关键差别是：并行任务通常提前定义，而这里的子任务由编排者根据当前输入临时决定。

例如修改一个大型代码仓库。你无法提前知道需要改几个文件，也不知道每个文件应该怎么改。编排者需要先分析问题，再决定搜索哪些模块和分配哪些任务。

这种模式适合复杂、可以拆分，但无法提前预测具体子任务的问题。编排者还要为每个执行者提供刚好够用的上下文，并定义统一的返回格式，否则汇总阶段会花大量成本重新理解零散结果。

第五种是评估与优化。

一个模型负责生成，另一个模型负责评价，再根据反馈继续修改。例如文学翻译中，生成模型负责翻译，评估模型检查语气、含义和术语，然后提出具体修改意见。

它成立有两个前提：结果确实能通过反馈改善，并且模型能够识别问题、给出有效反馈。循环还要有停止条件，例如评分达到阈值、连续两轮没有改善，或者预算已经耗尽，避免为了追求完美无限消耗。

如果没有清晰的评价标准，循环只会增加成本，不一定提高质量。

选择这五种模式时，可以先看不确定性出现在哪里：步骤固定，用提示链；输入类别不同，用路由；子任务彼此独立，用并行；子任务只能临时拆解，用编排者；结果能够被明确评价并反复改进，再使用评估优化。它们可以组合，但每增加一种模式，都应该对应一个清楚的问题。组合时也要保留可观测性，记录每一步的输入、输出、耗时和失败原因，否则流程越长，越难知道质量究竟在哪里提升或下降。

小结。

第一，先定位任务中的不确定性。

第二，用对应的 Workflow 处理这种不确定性。

第三，每增加一种模式都要解决一个明确问题。

### 时间轴自动生成｜自主 Agent

当任务的执行步骤无法提前确定，并且模型必须根据环境结果不断调整计划时，才真正进入 Agent 的范围。

Agent 的核心并不神秘，它是一个持续循环。

模型先理解目标，决定下一步行动，调用工具，从真实环境获得结果，再根据结果更新判断。这个循环一直持续到任务完成、遇到阻塞，或者触发停止条件。

这里最重要的不是模型思考了多久，而是每一步能不能获得真实反馈。

代码 Agent 修改文件后，可以运行测试；客服 Agent 执行退款后，可以读取接口结果；浏览器 Agent 点击按钮后，可以检查页面是否发生预期变化。

这些反馈才是 Agent 判断自己有没有取得进展的依据。

Agent 特别适合客服和编程，是因为这两类任务既需要对话和行动，又有相对明确的成功标准。

但自主性也会带来更高成本和错误累积风险。如果 Agent 每一步只有百分之九十的可靠性，连续执行十步后，全部正确的理论概率只有大约百分之三十五。真实系统当然可以通过检查和重试修正，但这个数字提醒我们：步骤越长，越不能只相信模型会自己走对。

因此实际系统需要沙箱、权限控制、最大迭代次数、清楚的停止条件，以及关键步骤的人工确认。还要记录每次工具调用和环境返回，保证失败后能定位、重放和恢复，而不是只看到一句任务失败。

Agent 不是放弃控制，而是把动态决策交给模型，同时用环境反馈和边界重新建立控制。判断能否放开自治，还要看失败是否容易恢复。读取信息、生成草稿这类动作容易回滚，可以给模型更大空间；付款、删除数据、对外发送等动作影响难以撤销，就应该缩小权限，并在执行前增加人工确认。最实用的做法，是按动作风险分级：低风险自动执行，中风险执行后检查，高风险执行前确认。

小结。

第一，行动之后必须读取真实反馈。

第二，失败风险会随着执行步骤累积。

第三，自治必须配合权限和停止条件。

### 时间轴自动生成｜工具设计

模型再聪明，如果工具难以理解，Agent 仍然会频繁失败。

人与软件之间需要设计界面，模型与工具之间同样需要设计清晰的 Agent-computer interface。

工具名称要明确，参数含义要清楚，还要说明输入格式、边界条件和常见示例。相似工具之间必须有明显区别。错误返回也要结构化地告诉模型发生了什么、哪些参数无效、是否允许重试，而不是只返回一个模糊的失败。

工具的返回格式也不要平白增加难度。例如让模型把大量代码塞进需要反复转义的 JSON，通常比直接返回代码块更容易出错。返回内容应该尽量短而明确，只保留下一步决策需要的信息；把完整日志全部塞回上下文，既增加成本，也会掩盖真正的错误。

Anthropic 在开发编程 Agent 时发现，模型使用相对路径容易犯错。把工具改成强制使用绝对路径后，问题便明显减少。

所以不要只优化提示词，还要观察模型怎样误用工具，再修改工具本身，让错误更难发生。上线前可以用缺参数、错误路径、重复调用和超时等输入专门测试工具接口，确认模型不仅在理想示例里会用，在异常情况下也能恢复。

小结。

第一，名称、参数和边界必须清楚。

第二，返回格式不要制造额外难度。

第三，要从接口层让常见错误更难发生。

### 时间轴自动生成｜选择方法

面对一个新任务，可以按照下面的顺序判断。

第一，单次模型调用，加上检索和示例，能不能解决？

第二，如果不能，任务能否拆成固定步骤？可以的话，优先选择提示链、路由或并行。

第三，如果任务可以拆分，但无法提前预测具体子任务，可以使用编排者与执行者。

第四，只有当整个执行路径都依赖环境反馈，并且模型必须持续自主决策时，才升级为 Agent。

框架可以帮助你快速完成模型调用、工具定义和流程连接，但也可能增加抽象层，让真实的提示词、模型输出和错误原因变得难以观察。可以使用框架，但必须理解它下面实际发生了什么。

上线前先保存最简单方案的基线，再比较升级后的任务完成率、平均成本、处理时长和人工介入次数。测试集要覆盖常见任务、困难任务和容易失败的边界任务，指标也要按任务类型拆开看，避免平均数掩盖关键场景的退化。如果复杂方案没有带来稳定提升，就应该退回简单版本。架构不是一次性选型，而是一轮可以测量的迭代。

小结。

第一，先验证单次模型调用能否完成任务。

第二，路径可控时优先选择合适的 Workflow。

第三，用可测量的结果决定是否升级为 Agent。

### 时间轴自动生成｜总结

最后，把整套方法记成三句话。

能用一次模型调用解决，就不要先做 Workflow。

能提前画出执行路径，就不要先做 Agent。

真正使用 Agent 时，要重点保证环境反馈、停止条件和工具接口可靠。

再把它压缩成一句话：从简单开始，用结果决定是否增加复杂度。真正成熟的 Agent 系统，不是让模型拥有最多自由，而是让每一层自由都有明确收益、真实反馈和可靠边界。

关注 Tiny Agent，用更少的时间理解 AI 和 Agent 领域真正重要的知识，把复杂概念变成能够判断、可以落地的方法，在快速变化中少走弯路，做出更有效的选择。
