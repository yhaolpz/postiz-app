Tiny Agent

什么任务值得交给 Agent？

很多人现在使用 AI，仍然像在使用一个更会写作的搜索框。
问一个问题，得到一段答案，真正的工作还是自己完成。
Agent 改变的是工作的基本单位。
你不再只问一个答案，而是把一段更长的任务交出去，
让它读取文件、调用工具、修改产物、检查结果，再持续推进到明确终点。
OpenAI 在二零二六年发布的 Codex 研究说明，
这种变化正在加速。
五月份，超过七成的抽样个人用户，
至少提交过一次估计相当于人类一小时以上工作量的请求；
四分之一提交过估计超过八小时的请求。
非开发者使用也增长很快。
但任务更长，不等于更适合委托。
这期视频会说明怎样区分真正适合 Agent 的工作，
怎样为任务准备上下文和验收方式，
以及什么时候该用聊天、AI 副驾、受监督 Agent，
或者可复用的委托流程。
最后你会得到一张任务升级判断表，
可以在交出研究、文档、表格、发布、运营或开发任务之前快速使用。
第一步，是不要再用消息数量衡量 AI 做了多少工作，
而要看它最终产出了什么成品，或者让真实系统发生了什么变化。
研究区分了咨询和委托生产。
咨询是提供信息或建议；
委托生产则要求系统生成文档、转换数据、修改系统、验证变更，
或者完成一整段流程。
它可以出现在很多任务中：根据资料起草备忘录，清理表格并核对总数，
排查一个失败的应用，准备一场活动，或者生成带引用的结构化研究报告。
任务时长的变化，说明用户开始相信 Agent 能接住更大的工作单元。
到二零二六年五月，
百分之八十点六的抽样个人用户提交过一次估计超过三十分钟人类工作量的请
求；百分之七十点二超过一小时；百分之二十五点六超过八小时。
而且一段对话里最宽、最复杂的要求，通常出现在第一轮。
后面的内容更多是纠偏和收窄。
所以高质量委托应该从一个完整的任务包开始，而不是只写一句模糊要求，
再用二十轮对话补齐漏掉的信息。
高频用户组织工作的方式也不同。
他们运行更长的任务，复用 Skill，
有时还会并行协调几个相互独立的 Agent。
人的角色从亲手输入每一步，
转向定义任务、观察进度、检查证据和决定何时介入。
这些数字需要保留边界。
人类完成时长是模型估计，只能作为方向性证据。
OpenAI 内部还具备更低的使用摩擦、更广的工具访问和更强的知识共
享，因此不能把内部采用速度直接当成每个团队都会发生的结果。
小结。
第一，看产出的成品或变化，不看消息数量。
第二，委托完整任务包，不要每轮纠偏才补一块关键上下文。
第三，把任务时长数据看成工作转变的证据，
不把所有长任务都自动交给 Agent。
一个任务很长，不代表它就是好 Agent 任务。
真正的问题，是这项工作有没有足够清楚的结构，
让 Agent 可以持续推进，而不是自己编造现实。
第一个条件是明确产物。
要求应该落到一个完成后的文件或可观察变化，
例如清理好的数据、通过测试的补丁、核对完成的报告，
或者等待人工确认的发布草稿。
如果只说帮我做营销，范围虽然很大，但完成标准完全看不见。
更好的任务会写清目标人群、资料来源、输出格式、限制条件，
以及这个结果最终要支持哪个决策。
第二个条件是可访问的上下文和工具。
Agent 必须拿到任务真正需要的文件、系统、参考资料和权限。
如果关键背景只存在于某个人脑中，或者藏在它无法访问的应用里，
这项工作就还不适合独立执行。
第三个条件是可信反馈。
代码任务可以跑测试，数据任务可以核对总数和结构，
浏览器任务可以确认页面是否发生预期变化，
研究任务可以附上引用并标出无法支持的结论。
没有反馈，Agent 可能一直前进，却离正确越来越远。
第四个条件是风险可控。
读取、起草和准备通常容易撤销；
发送消息、花钱、删除数据、修改生产环境和公开发布，出错后很难恢复。
这些动作需要更窄的权限和明确的人工确认点。
当任务同时具备明确产物、可访问上下文、可信反馈和可控风险时，
委托才从一次冒险，变成可以设计和管理的工作方式。
小结。
第一，定义明确的成品或可观察变化。
第二，提供判断进展需要的上下文、工具和反馈。
第三，增加自治之前先把风险边界写清楚。
任务升级判断表，会把刚才四个条件变成四道实际门槛。
第一道门槛，产物是否明确。
如果只是想得到解释、比较或灵感，就保留在聊天里；
如果需要一个完成后的成品，再继续向下判断。
第二道门槛，工作是否形成连续行动链。
Agent 要能从资料走到行动，再走到结果，
中间不需要反复等待隐藏信息。
如果链路被不可访问的系统或没有记录的决策打断，
就先把 AI 当成副驾，由人完成缺失步骤。
第三道门槛，推进过程是否产生证据。
每个重要步骤都应该返回会改变计划的信号，
例如测试结果、结构校验、来源引用、预览画面、核对总数或平台返回。
如果有证据但仍需要复杂判断，就使用受监督 Agent，并设置检查点。
第四道门槛，权限和后果是否可控。
容易撤销的工作可以自动继续；不可逆的工作必须在动作前停下，
请求人来确认。
Agent 很有用，并不意味着它需要无限权限。
这张表是根据研究证据整理出的实践工具，
不是从 OpenAI 原文复制出来的表格。
它的作用，是把从对话转向委托这件事，变成今天就能使用的判断。
先看表格清理。
产物是修正后的工作簿，输入已经提供，公式和总数可以作为反馈，
原文件也能保留，所以它通常可以升级到受监督或自动委托。
再看邮件发布。
资料整理、起草、排版、链接检查和预览都可以委托，但最终外发影响更大，
所以即使所有准备步骤通过，流程也应该停在发送按钮之前。
研究任务只有在资料可访问、观点能对应证据、不确定性能够暴露，
并且交付物有验收标准时才适合委托。
让 Agent 根据残缺来源直接决定高风险政策，
不会因为任务很长就自动变得安全。
因此四种工作模式很清楚。
需要答案，用聊天；行动链仍由人掌握，用 AI 副驾；
链路已经成立但需要检查点或专业判断，用受监督 Agent；
四道门槛都成立，而且流程足够稳定时，再使用可复用的委托工作流。
小结。
第一，一次只把工作模式升级一道门槛。
第二，不可逆动作必须保留明确的人工确认。
第三，选择能够稳定产出结果的最低自治模式。
任务通过判断表以后，下一步是给 Agent 足够的真实依据。
一个有效的上下文包包含五项：
目标、当前状态、相关文件或系统、限制条件，以及完成标准。
上下文不是越多越好，
最有价值的是能让 Agent 不靠猜测就行动的最小完整集合。
工具访问也要和任务匹配。
文档任务可能需要资料目录和输出路径；数据任务需要工作簿和校验规则；
发布任务需要有登录状态的页面、最终文案、封面，
以及停在发布按钮之前的明确边界。
然后设计反馈循环。
Agent 执行动作，读取真实结果，对照验收标准，
再决定继续、修复还是停止。
这才是 Agent 路径真正动态的原因。
计划可以因为证据而改变，不能因为模型自己感觉良好而改变。
测试失败、来源缺失、文件无效或平台拒绝，都应该成为新的决策，
而不是被隐藏的细节。
长任务还要有可见检查点。
保存中间产物，记录工具返回，保留失败原因，让当前状态可以恢复。
人应该能看清改了什么、还剩什么，
以及 Agent 为什么认为任务已经完成。
这套方法不只适用于代码。
招聘流程可以检查必填信息并在外联前停止；
财务流程可以核对总数并暴露异常；内容流程可以先检查尺寸、章节和链接，
再请求最终发布确认。
小结。
第一，准备包含目标、状态、上下文、限制和完成标准的最小完整任务包。
第二，让每个重要动作都返回能够改变计划的证据。
第三，保留检查点，让任务可以验收、继续和恢复。
委托不会消除人的判断，而是改变人的判断应该放在哪里。
读取、搜索、起草和准备这类低风险动作，通常可以给予更大的自治空间；
发送、付款、删除、修改生产环境和公开声明，则需要更窄的权限。
人工确认应该放在不可逆动作发生之前，
并且让 Agent 先准备好判断所需的证据。
不要让每一次无害读取都等待批准，也不要在破坏已经发生后才开始验收。
验收能力本身也是瓶颈。
如果 Agent 产出十倍内容，但结果仍然很难检查，
总体效率未必提升。
因此要在执行前设计验收标准。
可以要求差异报告、检查清单、核对结果、来源引用、测试记录或预览画面。
最好的验收产物，会让重要问题直接暴露，而不要求人重新拼出整个过程。
还可以在风险变化的地方拆分流程。
研究和起草可以自动进行，对外发送单独进入确认阶段；
数据转换可以自动完成，但删除源记录继续明确禁止。
为上下文缺失、连续失败、预算上限和结果含糊设置停止条件。
知道什么时候该停的 Agent，
比任何时候都声称完成的 Agent 更可靠。
小结。
第一，可撤销的准备工作可以扩大自治，不可逆行动必须缩小权限。
第二，执行前定义验收证据，让检查成本保持可控。
第三，上下文缺失、连续失败或完成状态含糊时必须停止。
最成熟的 Agent 使用者，不只是把更大的任务交出去，
还会把重复工作变成可复用流程。
OpenAI 的研究发现，
越来越多用户通过 Skill 和插件保存指令、工具访问和操作上下文。
到二零二六年六月，大约四分之一的活跃个人用户，
以及接近三分之一的活跃组织用户，至少调用过一个 Skill。
可以先找出那些目标、输入、限制和验收标准长期相似的重复任务。
然后把操作方法写进流程：上下文在哪里，应该使用哪些工具，
什么顺序不能改变，哪些检查必须通过，哪些动作明确禁止，
以及在哪里需要人工确认。
流程还要保留版本和证据。
发生失败以后，应该修改操作规则或工具接口，
而不是指望下一次 Agent 还能记得某一轮对话。
只有任务彼此独立，而且验收负担仍然可控时，并行才有价值。
三个 Agent 可以分别研究不同市场，或者检查不同模块；
但没有协同和归属规则时，不应该同时修改同一个产物。
人的角色会更像流程设计者和任务组合管理者：
选择工作、分配上下文、观察阻塞、验收高影响产物，
再从真实失败中改进可复用系统。
小结。
第一，把输入、限制和检查方式稳定重复的任务标准化。
第二，复用能够保存上下文、工具、权限和证据的操作流程。
第三，只有归属和验收清楚时，才并行执行相互独立的工作。
最后，把完整方法记成三句话。
第一，委托有明确产物的工作，不委托模糊主题。
第二，Agent 独立行动前，必须具备可访问上下文和可信反馈。
第三，限制权限，把人工确认放在不可逆动作之前。
任务升级判断表只会在条件成立时，
把工作从聊天升级为 AI 副驾、受监督 Agent，
再升级为委托工作流。
目标不是获得最大自治，而是在减少人工操作的同时，稳定产出并保留控制。

选择任务、准备上下文、验收结果、控制风险，
再把一次成功委托变成可重复的流程。
