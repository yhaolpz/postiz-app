任务通过判断表以后，下一步是给 Agent 足够的真实依据。

一个有效的上下文包包含五项：目标、当前状态、相关文件或系统、限制条件，以及完成标准。上下文不是越多越好，最有价值的是能让 Agent 不靠猜测就行动的最小完整集合。

工具访问也要和任务匹配。文档任务可能需要资料目录和输出路径；数据任务需要工作簿和校验规则；发布任务需要有登录状态的页面、最终文案、封面，以及停在发布按钮之前的明确边界。

然后设计反馈循环。Agent 执行动作，读取真实结果，对照验收标准，再决定继续、修复还是停止。

这才是 Agent 路径真正动态的原因。计划可以因为证据而改变，不能因为模型自己感觉良好而改变。测试失败、来源缺失、文件无效或平台拒绝，都应该成为新的决策，而不是被隐藏的细节。

长任务还要有可见检查点。保存中间产物，记录工具返回，保留失败原因，让当前状态可以恢复。人应该能看清改了什么、还剩什么，以及 Agent 为什么认为任务已经完成。

这套方法不只适用于代码。招聘流程可以检查必填信息并在外联前停止；财务流程可以核对总数并暴露异常；内容流程可以先检查尺寸、章节和链接，再请求最终发布确认。

小结。

第一，准备包含目标、状态、上下文、限制和完成标准的最小完整任务包。

第二，让每个重要动作都返回能够改变计划的证据。

第三，保留检查点，让任务可以验收、继续和恢复。
