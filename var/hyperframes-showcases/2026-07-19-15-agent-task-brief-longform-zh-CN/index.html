Agent 任务委托单

很多 Agent 任务失败，并不是因为提示词不够聪明，
而是因为委托从一个空白输入框开始，
真实工作却散落在文档、消息、表格、既有决定和没有说出口的标准里。
OpenAI 关于日常工作的建议，起点完全不同：
把团队已经在使用的材料交给 Agent，
再要求它先产出一份真正可用的具体产物。
目标不是聊得漂亮，而是得到一份可以检查、修改和继续推进的工作结果。
这会改变一份好委托需要包含的内容。
Agent 要看到源材料、约束、审阅预期和交付格式，
也要明确知道最后需要带回什么产物。
这条视频会把这些要素整理成一张 Agent 任务委托单。
你会学会怎样先定义产物、打包必要上下文、写清边界、让审阅标准可观察，
并指定能够顺利交接的输出格式。
先记住一个原则：不要要求 Agent 理解你的整个世界。
只提供生成第一版可审阅产物所需的最小完整信息包。
委托要从产物开始，而不是从活动开始。
帮我推进这个项目”只描述了投入；
“比较三个发布方案并生成一份决策备忘录”，
才描述了一个别人可以审阅和使用的结果。
OpenAI 给出的日常工作例子包括简报、摘要、演示文稿、工作簿、计
划、流程文档、每日简报、周度更新、发布材料包、财务复盘和流程审计。
这些产物都有清楚的形态，也都承接一个下一步决定。
用可操作的方式定义产物：谁会使用，它支持什么决策或动作，
达到什么状态就算第一版可用。
第一版不必完美，但必须完整到足以暴露缺失事实和薄弱推理。
把交付物和更大的目标分开。
提高客户留存”是目标；
一份包含分群分析、三个优先假设、支撑证据和负责人建议的文档，
才是可以推动目标的产物。
可以使用“动作、对象、决定”这个结构。
生成周度运营简报，让负责人识别阻塞；制作发布检查单，
让发布负责人决定继续还是停止；把分散研究整理成对比表，
让采购者能够选择。
最常见的错误，是只要求 Agent 动起来，却没有定义完成状态。
它会搜索、总结和写作，但双方都不知道什么时候算结束。
明确产物，才能让后面的约束和审阅标准有落点。
小结。
第一，描述工作之前，先定义一个具体产物。
第二，把产物连接到真实使用者和真实决定。
第三，让第一版足够可用，能够暴露还缺什么。
真实工作通常已经有上下文，
可能存在日历、消息、邮件、文档、看板、电子表格、跟踪器、演示稿或讨论
记录里。
任务委托单要指向那些真正会改变结果的材料。
不要把所有内容一股脑塞进去，而是建立源材料地图。
对每个来源说明它包含什么、为什么重要、更新时间，
以及它是一手事实、支撑材料还是背景信息。
把权威材料放在前面。
正式需求高于聊天摘要，当前看板高于上季度截图，
实现问题优先查看仓库文件，而不是复制出来的一小段代码。
两个来源冲突时，Agent 必须知道谁优先。
不要掩盖不确定性。
把假设、缺失记录、过期数字和仍待确认的事实标出来。
这样第一版产物才能区分证据和推断，而不是把两者抹平成一个自信结论。
例如制作周度运营简报，
源材料可以包括当前项目跟踪器、会议决定、故障记录和未来截止日期。
它不需要团队历史上发过的每一条消息。
最常见的错误是上下文堆积。
附件越多，越可能埋没信号、制造矛盾并拉长审阅。
只有当一个来源会影响结论、约束、决定或必需产物时，才应该进入信息包。
小结。
第一，只提供会改变产物的源材料。
第二，标明哪些来源最新、最权威。
第三，显式标注缺失事实和不确定性，不让 Agent 猜。
约束定义了产物可以在什么范围内发挥作用。
它包括截止时间、受众、范围、预算、权限、禁止动作、必须包含的说法，
以及绝对不能出现的内容。
把约束写成可以检查的测试，不要停留在模糊偏好。
尽量简洁”很难验收；“控制在一页以内，最多五条建议，
把未决问题单列”就能直接检查。
区分硬约束和偏好。
硬约束不满足就不能接收，
例如法律文案、数据边界、批准渠道、最高预算和必需格式。
偏好用于指导质量，在证据发生冲突时可以重新取舍。
不仅要写内容边界，也要写动作边界。
Agent 可以读取文件并起草邮件，不代表可以直接发送；
可以准备发布材料，不代表可以公开发布。
动作越难撤回，授权门槛就应该越高。
同时定义停止条件。
关键来源缺失、两个权威记录冲突、成本超过上限、权限不清楚，
或者任务扩大到原委托之外时，都应该暂停。
好的暂停会带回阻塞原因和继续所需的最小决定。
最常见的错误，是把约束留在委托人的脑子里。
Agent 会用看起来合理的默认值补空白，
但对真实受众、政策和后果来说，这些默认值可能完全不可接受。
小结。
第一，把重要偏好转换成可检查的限制。
第二，区分硬约束和可协商的指导。
第三，执行前写清权限和停止条件。
审阅预期告诉 Agent，第一版会按什么标准被判断。
没有这些标准，Agent 往往优化语言流畅和形式完整，
却忽略真正影响决定的部分。
生产之前先定义验收条件：产物是否回答真实问题，是否使用必需证据，
是否遵守约束，是否区分事实和推断，是否把未解决事项明确留下。
让证据可以被检查。
可以要求标注材料出处、保留计算说明、列出变更文件、提供截图、测试结果
或简短决策记录。
具体证据随产物变化，但必须让审阅者能够验证关键结论。
分清审阅角色。
Agent 可以检查结构、一致性、缺失字段和常规验证；
领域专家判断专业结论；
承担责任的人批准公开、发送、花钱、删除或做出受保护声明等高后果动作。
把第一版可用产物当作审阅界面。
不要等到一份精美终稿完成后，才发现目标理解错了。
早期审阅应该尽快暴露未决问题、来源冲突和假设，因为此时修改成本最低。
最常见的错误，是要求“高质量”，却没有定义拒绝规则。
如果没人能说明为什么失败，Agent 就无法瞄准真正标准，
审阅者也无法给出精确反馈。
小结。
第一，Agent 开始前先定义验收条件。
第二，要求能够检查关键结论的证据。
第三，让承担责任的人继续批准高后果动作。
交付格式不是装饰。
它决定 Agent 完成后，工作怎样被消费、编辑、分享或执行。
内容有用但容器错误，只会制造另一项人工整理任务。
写清最终载体和结构。
产物是一份 Markdown 简报、一页文档、带公式的电子表格、演示
文稿、检查清单、工单更新，还是可以直接放进仓库的一组文件？
指定必需章节、字段、顺序、尺寸、命名规则和保存位置。
如果下游系统要解析结果，就提供结构定义或示例；如果由人审阅，
就优化扫读速度和决策效率。
不仅交付产物，也要交付当前状态。
状态应该说明改了什么、检查了哪些证据、还剩什么未决问题，
以及下一步允许执行什么。
任务可能在另一个会话恢复时，这一点尤其重要。
例如发布材料包，
可能需要发布检查单、批准后的文案、媒体路径、风险表和最终继续或停止状
态。
只写一段讨论发布思路的文字，不能算完成。
最常见的错误，是把格式放到最后再想。
Agent 解决了认知问题，
却返回一个无法发布、导入、审阅或恢复的形态，最终仍需要大量返工。
小结。
第一，明确交付载体和必需结构。
第二，必要时写清命名、位置和机器可读规则。
第三，把产物和已验证的交接状态一起交付。
现在把前面的内容合并成一张 Agent 任务委托单。
它是工作负责人和负责生成第一版产物的 Agent 之间的一份紧凑契约
。
第一栏是目标产物：写清交付物、使用者、支持的决定，
以及什么状态算可用。
第二栏是源材料地图：列出批准输入、权威顺序、更新时间和已知缺口。
第三栏是约束：硬限制、偏好、权限、禁止动作和停止条件。
第四栏是审阅：验收标准、必需证据、人工审阅者，
以及哪些动作必须由人批准。
第五栏是交付格式：必需章节、结构定义、尺寸、文件名和输出位置。
第六栏是交接：已完成工作、未决问题、核对证据和下一步允许动作。
执行前使用这张委托单，任务变化时也要同步更新。
新来源改变结论，就记录来源；审阅者改变标准，就更新验收条件。
委托单描述的是当前任务，不是保存已经失效的旧指令。
最常见的错误，是把委托单写成一份仪式化提示词。
每个字段都应该改变 Agent 的执行方式或结果的审阅方式。
两者都不会改变的句子，应该删除。
小结。
第一，把产物、来源、约束、审阅、格式和交接打包。
第二，事实或验收条件变化时，及时更新委托单。
第三，删除任何不会改变执行或审阅的指令。
高质量 Agent 工作不是从空白提示词开始，
而是从真实材料和一个具体产物开始，让别人能够检查、修改并继续推进。
第一，先定义产物，再提供最小完整源材料包。
第二，让约束、审阅预期和交付格式可观察。
第三，返回一份可审阅的第一版，并附带证据、未决问题和清楚交接。
下一次委托时，直接使用 Agent 任务委托单。
如果产物、来源、约束、审阅、格式或交接仍然模糊，就先修复那一栏，
再给 Agent 更多工具和权限。
