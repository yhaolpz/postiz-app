AI 工作模式怎么选？

很多 AI 工作流出错，不是提示词不够好，
而是一开始就选错了人与 AI 的关系。
需要一起打磨时只问了一个问题，已经可以执行时还在聊天，
或者目标没想清楚就交给 Agent。
真正有用的问题不是“AI 最多能替我做多少”，
而是“这个阶段需要多少人类判断，
又能安全交给 Agent 多少执行”。
这是两个独立决定。
微软 2026 年工作趋势指数提出四种模式：提问、协作、探索和委派。
它们不是成熟度阶梯，每种模式对应不同的任务条件。
接下来做出一张可复用的四模式地图：
每种模式何时使用、什么该交出去、什么必须留下，
以及条件变化时如何切换。
先记住一条底线：执行可以交给 Agent，但结果责任仍然属于人。
好的工作流会把这条边界变得更清楚。
先画两条彼此独立的轴。
第一条是人类参与强度，
表示任务需要多少持续判断、品味、沟通和只有你掌握的上下文。
第二条是 Agent 执行强度，
表示 AI 能承担多少搜索、整理、转换、协调和工具操作。
这两条轴不是此消彼长。
同一个任务完全可能同时需要很高的人类参与和很高的 Agent 执行。
比如一次产品发布，人要持续做取舍，
Agent 同时可以收集证据、更新文档、检查依赖和执行重复步骤。
然后问四个路由问题：想要的结果是否已经明确？
过程里必须保留多少人类判断？
成功能不能被可观察的证据验证？
如果 AI 做错，后果有多大？
这些问题比“模型会不会做”更重要。
能力只是输入之一。
就算模型有能力，当目标含糊、证据薄弱或者错误代价很大时，
它也不适合直接接管执行。
微软的调研把这条人类边界说得很清楚。
百分之八十六的受访者把 AI 输出当作起点，
同时仍然认为最终结果由自己负责。
关键不是移除人，而是把人的注意力放在最能改变结果的地方。
所以，先画地图，再选工具。
用一条轴表示人类参与，用另一条轴表示 Agent 执行，
判断的是任务的当前阶段，而不是给整个项目贴一个永远不变的标签。
第一，人类参与强度和 Agent 执行强度彼此独立。
第二，根据任务条件选择，不要根据自动化野心选择。
第三，结果责任始终留在人手里。
当一个边界清楚的不确定点卡住了人的决定，就使用提问模式。
你没有把工作交出去，只是在购买一小块清晰度：
一个定义、一组比较、一个反例、一次计算，或者一段简洁解释。
一个好问题应该说明它要支持什么决定。
不要只问“有哪些好的新手引导”，
可以改成“一个只有一个核心动作的移动工具，最适合哪种新手引导？
出现什么失败信号时应该放弃它？
输出要小到能够检查。
可以要求 AI 说明前提、置信边界或者证据来源。
如果答案无法验证，就不能让它悄悄变成下一步行动的基础。
提问也适合嵌入正在进行的工作。
开发者可以问一条报错意味着什么，作者可以要三种不同的叙事角度，
负责人可以问哪个约束带来最大的取舍。
操作人依然是人。
最常见的失败是问题悄悄膨胀。
一个简单问题扩成方案，再扩成产物，最后变成了没人明确授权的动作。
要及时停止。
如果真正需要的是执行，就明确切换模式，并重新定义边界。
提问模式的退出条件是做出决定。
一旦不确定点被消除，就回到人的任务，或者进入协作。
无休止的追问，往往说明问题已经不再是一个单点问题。
第一，用提问消除一个边界清楚的不确定点。
第二，让问题始终服务于一个具体决定。
第三，在答案影响行动之前完成验证。
当目标很重要，但到达目标的路径需要人持续判断，就使用协作模式。
人与 AI 都保持活跃。
AI 提议、转换、检查，人负责选择、纠偏，
并补充系统无法自行推断的上下文。
协作最好围绕同一个真实产物展开。
直接编辑那份简报、代码、模型、文案或分镜。
具体产物能把抽象的品味变成可见决定，
也能避免对话退化成一串彼此无关的建议。
使用短循环。
一次只做一个有意义的改动，检查结果，再说明哪里变好、哪里仍然不对。
下一条指令应该更新产物和验收条件，而不是每次都从空白提示词重新开始。
写作、设计、诊断、规划和代码评审都适合协作，因为质量往往来自对比。
AI 可以快速产生选项，但哪些取舍属于最终结果，仍然由人决定。
最常见的失败是无形收敛。
对话看起来很热闹，但没人说得清什么叫完成。
保留一张很小的评分卡：读者、目的、约束、不能丢的事实，
以及判断产物是否合格的测试。
当剩余步骤变得机械、可检查时，协作就可以结束，
把后续工作打包成委派任务。
如果产物仍有相互竞争的方向，就继续协作，或者短暂回到探索。
第一，当每轮迭代都需要人类判断时使用协作。
第二，围绕真实产物协作，不要只做抽象聊天。
第三，让验收条件始终可见。
当目的地本身还不稳定，就使用探索模式。
你可能不知道真正重要的问题是什么，不知道哪个解释成立，
也不知道哪个机会值得投入。
此时目标不是立刻生产，而是把地图画得更准确。
让 AI 扩大并整理问题空间。
生成互相竞争的假设，寻找反证，归类未知项，比较不同用户，
或者揭示依赖关系。
好的探索会制造有意义的对比，而不是只把清单变得更长。
人的参与强度依然很高，因为相关性来自具体情境。
Agent 可以呈现模式，
但哪些信号重要、哪些来源可信、哪个问题值得投入下一小时，
仍然需要人判断。
好奇心可以被增强，不能被外包。
可以使用四列证据板：我们现在相信什么，已经观察到什么，
什么证据会让我们改变判断，下一步准备测试什么。
这样就不会把生成的可能性和已经验证的事实混在一起。
最常见的失败是“可能性表演”。
想法越多，越容易产生进展感，同时真正的决定却不断后退。
给探索设置时间、来源或实验预算，并提前写清它最终要支持哪个决定。
探索应该以一个更窄的问题、一个选定方向或者一次测试结束。
然后可以切到提问补齐事实，切到协作打磨方案，或者切到委派执行测试。
第一，在目的地还不稳定时先探索。
第二，生成对比、证据和更好的问题。
第三，用一个决定结束探索。
当结果已经明确、可用工具能够执行步骤，而且成功可以被检查时，
就使用委派模式。
人定义任务包，Agent 运行工作，并带着证据回来。
一个完整的委派任务包有六部分：
目标结果、相关上下文、硬约束、允许使用的工具、停止条件，
以及交接时必须返回的证据。
少了任何一项，都会把隐藏的解释工作推给 Agent。
给出的权限要与错误后果匹配。
读取文件、起草报告、运行测试通常可逆；
发布内容、花钱、删除数据、联系他人则需要明确闸门。
自主性是被限定的权限，不是 Agent 的性格。
要求可观察证据。
代码任务返回改了哪些文件和测试结果；
研究任务返回一手来源和未解决的不确定项；
发布任务返回公开地址、可见状态和所在频道。
最常见的失败，是审查 Agent 的讲述，而不是审查结果。
它可以说得非常自信，产物却仍然不完整。
应该检查实际输出、验证结果和边界条件。
自信不是证据。
委派以接受、拒绝或者切换模式结束。
如果 Agent 发现需求含糊，就把任务带回协作；如果发现未知问题，
就转入探索。
能在正确位置停下来，本身就是高质量交接的一部分。
第一，只委派一个已经明确的任务包。
第二，给出工具、约束、停止条件和证据要求。
第三，审查结果，不要审查 Agent 的自信。
真实工作很少从头到尾只用一种模式。
一个项目可能从探索开始，用提问消除一个不确定点，进入协作打磨产物，
最后再把可重复的执行交给 Agent。
从能够产生下一个有效结果的最低自主模式开始。
这不是害怕自动化，而是在扩大执行前先学清任务边界。
证据越充分，才越适合增加自主范围。
观察四个切换信号：目标结果变得更清楚；过程中出现新的判断点；
证据与原计划冲突；或者错误后果发生变化。
任何一个信号，都会改变合适的人机平衡。
比如准备一次客户发布。
先探索受众和风险，再用提问补齐一个比较，协作完成定位与验收标准，
把素材检查和定时发布交给 Agent，
最终在正式发布闸门回到人工确认。
每次交接只记录一行：
当前模式、下一个结果、负责人、验证证据和停止条件。
这个小日志能阻止权限悄悄扩张，也能让被中断的任务更容易恢复。
整个循环只有四步：选择、运行、检查、重新路由。
不要奖励 Agent 一直保持自主，
要奖励整个工作流得到正确、有用并且责任清楚的结果。
第一，从能够完成任务的最低自主模式开始。
第二，当不确定性或错误后果变化时切换。
第三，在扩大自主范围之前先记录证据。
四模式地图不是需要背诵的分类，而是一件路由工具。
下一次写提示词前，
先指出不确定性是什么、哪些判断必须留在人手里、Agent 能接走哪些
执行，以及什么证据能证明结果成立。
第一，把人类参与强度和 Agent 执行强度分开。
第二，根据不确定性、判断密度和风险匹配模式。
第三，有意识地切换，并保留结果责任。
一个单点不确定性用提问，需要迭代判断用协作，目的地不稳定先探索，
结果明确且可验证再委派。
任务条件变了，模式也应该跟着变。
持续关注 AI 和 Agent 的工作方法很有价值，
因为模型会不断变化，但定义目标、边界、证据和责任的需求不会消失。
掌握这些能力，才能使用更强的系统，同时不交出自己的判断。
面对下一个真实任务，先画两条轴，选择能产生有效结果的最小模式，
并在开始前写下停止条件。
这样，AI 工作流才能同时变得更强、更可靠。
